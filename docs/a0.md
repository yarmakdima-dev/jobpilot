# A0 — Company Research Agent

**Status:** Real backend (shipped 2026-04-28)
**Stack:** Gemini 2.5 Pro + google_search grounding + responseSchema
**Entry-point:** `orchestrator/a0.py::run_a0(role)`
**Implementation:** `agents/a0/research.py::research_company(role)`

---

## What A0 does

Takes a role record that has cleared F1 and produces a populated
`companies/<domain>.json` matching `company_profile_schema v0.2`.

Single Gemini call: grounding search + structured JSON output in one pass.
No two-step formatting pipeline.

Output includes: company snapshot, business model, strategy, leadership
(with `war_position` and `parallel_business_flag` for all leaders),
market view, hiring signal, risks, judgment-call gates, and source citations.

Does **not** populate `360_synthesis` — that is F2's lane.

## Auth

```bash
export GEMINI_API_KEY="your_key_here"
```

Add to `~/.zshrc` for persistence. The key must have access to Gemini 2.5 Pro
via Google AI Studio. Free tier: 1,500 grounded requests/day.

## Cost estimate (free tier)

| Call type | Tokens (typical) | Free-tier daily budget |
|-----------|-----------------|------------------------|
| First pass (grounded) | ~2,000–4,000 | 1,500 requests/day |
| Fallback pass | ~2,000–4,000 | counted against same budget |

At 1–2 roles/day, free tier is sufficient. Token usage is logged to
`decisions.log` per call under `tokens_used`.

**Rate limit behaviour (Q1=B):** On 429 / RESOURCE_EXHAUSTED, the client
retries with exponential backoff (2s, 4s, 8s) up to 3 attempts. Assumes
billing is enabled on the account. After 3 failures, raises `A0ResearchError`
and the role moves to `research_failed` state (hold; surfaces in daily report).

---

## Fallback behaviour

If grounding returns fewer than 3 sources on the first pass:

1. The first-pass source URLs are injected explicitly into a second prompt.
2. A second Gemini call is made with the enriched prompt.
3. If the first pass returned **zero** sources, `A0NoSourcesError` is raised
   immediately (no fallback — nothing to inject).

Fallback trigger is logged to `decisions.log` as `fallback_triggered: true`.

---

## Multilingual research (Q2=A)

Gemini's native grounding handles multilingual search. No explicit
Russian-language search terms are injected in v1. If integration testing
reveals thin coverage for RU/BY-origin leader war-position research, add
explicit terms in a future rubric update.

## Judgment-call detection

After Gemini returns the profile, `orchestrator/a0.py` runs its own
judgment-call scan (authoritative pass — overwrites what Gemini populated):

- **`russian_or_belarusian_market_or_business`** fires when:
  - A leader with RU/BY origin has `war_position.value` of `silent`,
    `not_researched`, or null.
  - A leader has an active parallel business with `carve_out_escalation=True`.

When this gate fires, the role gets `gate_needs_judgment_call` set and the
orchestrator holds downstream dispatch (no F2 until resolved or overridden).

Explicit `anti_war` clears the gate. Explicit `pro_war` or `accommodative`
is a hard disqualifier (handled by F2 rubric, not A0).

## Known limitations

- **Source index is capped at S1–S8** in the responseSchema. Gemini may cite
  more sources in grounding metadata; extras are backfilled up to index 10.
- **Grounding quality varies** by company size and public indexing. Obscure or
  newly-founded companies may trigger the thin-sources fallback frequently.
- **RU/BY language coverage** relies on Gemini's native multilingual grounding.
  Quality for Russian-language sources (Meduza, VK, company sites) has not
  been benchmarked — flag gaps via `gate_needs_judgment_call` and probe in call.
- **`responseSchema` + `google_search`** are combined in a single call.
  If a future Gemini API version separates these features, `gemini_client.py`
  is the only file to update.
- **Gemini field drift is real.** Even with `responseSchema`, grounded responses
  may return semantically-correct fields under adjacent names such as
  `headquarters_city`, `founding_year`, `revenue_streams`, or `pricing_model`.
  `agents/a0/research.py` normalizes these variants before validation.
- **`360_synthesis` is never populated by A0** — that field is F2's exclusive
  lane. Any Gemini output in that field is stripped before writing.
- **Research gaps are preserved explicitly.** If a grounded run fails to resolve
  a single field like HQ while the rest of the profile is strong, A0 marks the
  gap in `instance_meta.confidence_notes` instead of discarding the whole
  profile.

## Wiring into the pipeline

The state machine in `orchestrator/state_machine.py` calls `run_a0(role)`
when a role transitions from `f1_passed` → `researching` → `researched`.
No changes to `state_machine.py` were needed — the existing `_a0_handler`
already delegates to `run_a0`, which now calls the real backend.

To plug in a different research provider later: replace `_call_perplexity`
and `_call_formatter` stubs (gone) with a new implementation in
`agents/a0/research.py`. The orchestrator layer stays untouched.
